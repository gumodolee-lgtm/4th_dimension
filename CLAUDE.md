# CLAUDE.md — 프로젝트 공통 규칙 (4가지 시스템)

## CRITICAL: 기본 규칙

### Rule 1 — 정확성
- 제공된 맥락(파일, 대화 내용) 내에서만 정보를 사용하라.
- 정보가 부족하면 추측하지 말고 "정보가 부족합니다"라고 답하라.

### Rule 2 — 코딩 스타일 준수
- 기존 코딩 스타일, 폴더 구조, 사용 중인 라이브러리 패턴을 철저히 준수하라.
- 내가 명시하지 않은 새로운 라이브러리를 임의로 도입하지 마라.
- 코드를 수정하기 전에 해당 파일을 먼저 읽고 이해하라.

### Rule 3 — 단계별 사고
- 복잡한 작업 전에 단계별 논리 과정을 먼저 정리하라.

## 시스템 1: 자동 매뉴얼

### 매뉴얼 참조 규칙
- 항상 `docs/MANUAL_*.md` 파일들을 최우선으로 참고하라.
- 작업 요청이 있을 때마다:
  1. 요청 내용의 **키워드** 확인 (API, DB, auth, UI 등)
  2. 수정할 **파일 경로** 확인 (backend/**, frontend/**, pages/** 등)
  3. 파일 안의 **프레임워크/라이브러리** 패턴 확인
  4. 위 조건에 해당하는 매뉴얼을 스스로 읽고 규칙을 따르라.

### 매뉴얼 충돌 시
- 사용자 요청이 매뉴얼과 충돌하면, 그대로 수행하지 말고 충돌 부분을 먼저 설명하고 대안을 제안하라.

### 매뉴얼 목록
- `docs/MANUAL_OVERVIEW.md` — 전체 개요 및 매뉴얼 색인
- `docs/MANUAL_SCRIPTS.md` — 스크립트 규칙 (API 호출, 분류, 배치 처리, 에러 처리)
- `docs/MANUAL_PROMPTS.md` — 프롬프트 규칙 (템플릿, 분류 일관성, 버전 관리)
- `docs/MANUAL_DOCX.md` — DOCX 생성 규칙 (폰트, 색상, 구조, 템플릿)

## 시스템 2: 작업 기억 (계획·맥락·체크리스트)

### 문서 구조
각 기능/이슈마다 아래 세 문서를 항상 유지한다:
- **PLAN**: `docs/feature_<name>_plan.md` — 목표, 요구사항, 큰 단계
- **CONTEXT**: `docs/feature_<name>_context.md` — 설계 이유, 관련 이슈, 연관 시스템
- **TODO**: `docs/feature_<name>_todo.md` — 할 일 체크리스트

### 규칙
1. **새 기능 요청** → 먼저 세 문서의 초안을 제안하고 사용자 승인을 받는다.
2. **작업 진행** → PLAN, CONTEXT, TODO를 먼저 읽고, TODO에서 '현재 해야 할 일' 1~2개만 작업한다.
3. **대화 종료 전** → 항상 TODO를 현재 상태로 업데이트하는 제안을 한다.
4. **새 대화 시작** → 기존 문서들을 먼저 읽고 이어서 작업한다.
5. **한 번에 "다 해줘" 금지** → 작은 단위로 나눠서 진행한다.

## 시스템 3: 자동 품질 검사

### 완료 전 필수 체크
어떤 작업이든 "완료"라고 말하기 전에 스스로 아래를 수행한다:

1. **수정 파일 목록** 출력
2. **셀프 리뷰** 수행:
   - [ ] 에러 처리 누락 여부
   - [ ] 보안상 취약점 (XSS, SQL injection, 환경변수 노출 등)
   - [ ] 명백한 버그 가능성
   - [ ] 타입/null 안전성
   - [ ] 엣지 케이스 처리
3. **문제 발견 시** → 코드를 고쳐서 바로 반영
4. **테스트 제안** → 가능한 경우 테스트 실행 명령어를 제안

### 프로젝트 전용 품질 검사
- [ ] API rate limit 준수 (2초 간격, 배치 5개)
- [ ] 분류 결과의 4요소(생각/믿음/꿈/말) 매핑 정확성
- [ ] DOCX 출력 시 한글 폰트(맑은 고딕) 설정 확인
- [ ] 프롬프트 템플릿 변경 시 기존 분류 결과와의 일관성

### 테스트/린트 명령어 (프로젝트에 맞게 수정)
```bash
# JavaScript/TypeScript
npm test
npm run lint
npx tsc --noEmit

# Python
pytest
mypy .
flake8 .
```

## 시스템 4: 역할 분리 (전문 에이전트)

### 역할 정의
상황에 따라 아래 역할 중 하나로 행동한다:

| 역할 | 담당 |
|------|------|
| **Planner** | 요구사항 분석, 설계/계획 문서 작성 |
| **Implementer** | 코드 구현 및 리팩터링 |
| **Reviewer** | 코드 리뷰, 구조·안전성 피드백 |
| **QA** | 테스트 케이스 작성, 경계 상황 점검 |

### 규칙
- 사용자 요청이 모호하면: "이번에는 어떤 역할로 도와줄까요?"라고 먼저 물어본다.
- 가능하면 **Planner → Implementer → Reviewer → QA** 순서의 흐름을 제안한다.
- 큰 작업은 역할을 명시하고 순서대로 진행한다.

## 자주 쓰는 프롬프트 (참고용)

```
지금부터 새로운 기능을 시작할 거야.
1) 이 기능에 대한 PLAN/CONTEXT/TODO 세 문서를 제안해 줘.
2) 이미 있다면 docs/feature_* 문서를 찾아서 읽고 현재 상태를 요약해 줘.
3) 지금 단계에서 내가 해야 할 일 1~2개만 제안해 줘.
```

```
지금 관련된 PLAN/CONTEXT/TODO 문서들을 먼저 읽고,

작업 후에는 TODO 문서를 어떻게 업데이트할지 제안해 줘.
```

```
지금까지 수정한 파일들을 기준으로:
1) 에러 처리 누락
2) 보안상 취약점
3) 명백한 버그 가능성
4) 테스트 부족
을 스스로 점검해 줘. 필요한 수정이 있으면 코드까지 반영해 줘.
```
